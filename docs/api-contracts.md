# API Contracts

This document describes the currently implemented mobile API contracts used by the `mobile-react-native` repository.

## Transport

All requests are sent as JSON over HTTP using Content-Type: application/json.

Default request method:
POST

Default timeout:
30000 ms

Base URL:

- provided by `EXPO_PUBLIC_BASE_URL`
- local development may use `http://localhost:<port>` or a local LAN host
- non-development builds must use `https://`

Client behavior:

- non-2xx responses are treated as errors
- timeout raises "Request timeout"

---

## API Groups

The mobile client currently uses these API groups:

auth

- /api/auth/qr/scanned
- /api/auth/qr/manual-code
- /api/auth/qr/confirm

authMobile

- /api/mobile/auth/start
- /api/mobile/auth/confirm

services

- /api/mobile/services/sync

# 1. QR Authorization

## 1.1 Scan QR Request

Endpoint:
POST /api/auth/qr/scanned

Purpose:
Used after scanning QR and sending device identity.

Request body:
{
"requestId": "string",
"nonce": "string",
"publicKey": "string",
"appInstanceId": "string"
}

Success response body:
{
"ok": true,
"data": {
"payload": {
"requestId": "string",
"clientId": "string",
"projectId": "string",
"brandColor": "string optional",
"description": "string optional",
"challenge": "string",
"service": "string",
"location": "string",
"expires": 0
},
"signature": "string",
"key_id": "string"
}
}

## 1.3 Confirm QR

Endpoint:
POST /api/auth/qr/confirm

Request body:
{
"approved": "string",
"requestId": "string",
"challengeSignature": "string",
"nonce": "string",
"publicKey": "string",
"appInstanceId": "string"
}

Current client behavior:

- approve requires a valid `challengeSignature` generated by the device
- deny does not require a signature and is processed as an explicit rejection
- `challengeSignature` is optional for deny and, if present, is ignored by the backend

Success response body:
{
"ok": true,
"data": {
"payload": {
"accessGranted": true,
"deviceAccessPass": {}
},
"signature": "string",
"key_id": "string"
}
}

Request body:

"public_key": "string",

"client_id": "string",
"service_name": "string or null",
"finish_token": "string"
},
"signature": "string",
"key_id": "string"
}
}

# 3. Services Sync

Endpoint:
POST /api/mobile/services/sync

Request body:
{
"currentVersion": 0,
"cursor": "string or null"
}

Success response body:
{
"ok": true,
"data": {
"payload": {
"version": 0,
"changed": true,
"upserts": [],
"deletes": [],
"nextCursor": "string or null",
"hasMore": true
},
"signature": "string",
"key_id": "string"
}
}

# 4. Notes

- QR auth uses camelCase
- Mobile auth uses snake_case
- This is acceptable for now

---

# 5. Error Handling

- Non-2xx → Error("HTTP <status>")
- Timeout → Error("REQUEST_TIMEOUT")

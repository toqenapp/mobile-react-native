# QR Format

This document describes the QR request format used by the Toqen Mobile application.

The QR mechanism is designed for secure, real-time authorization and must never expose secrets or reusable access data.

---

## Purpose

The QR code is used to transfer a short-lived authorization request from a service to the mobile device.

It is not used to transmit credentials, long-lived tokens, or any secret material.

Its role is to provide enough context for the mobile app to initiate a secure challenge-response flow with the backend.

---

## Security Goals

The QR format is designed to support the following goals:

- no secrets inside QR
- no reusable login artifacts
- short-lived request context
- resistance to replay
- safe interception characteristics
- compatibility with device-bound signature flows

---

## Design Rules

A valid Toqen QR must be:

- temporary
- single-use
- challenge-based
- verifiable by the backend
- safe to display publicly for a short period of time

A valid Toqen QR must NOT contain:

- passwords
- session tokens
- refresh tokens
- private keys
- access passes
- reusable bearer credentials

---

## Recommended URI Format

Recommended structure:

toqenapp://auth?request_id=<request_id>&nonce=<nonce>&expires=<unix_ts>&service=<service_name>

Example:

toqenapp://auth?request_id=91f2d&nonce=8fa92c1a&expires=1710000000&service=Example%20Service

This is the current mobile-readable format aligned with the implemented parser.

---

## Core Fields

### request_id

Type:
string

Description:
Unique authorization request identifier generated by the backend.

Requirements:

- unique per authorization request
- server-resolvable
- single-use

---

### nonce

Type:
string

Description:
Short-lived random value associated with the request.

Purpose:

- replay protection
- request binding
- flow uniqueness

Requirements:

- unpredictable
- unique per request
- validated by backend

---

### expires

Type:
number

Description:
Authorization request expiration timestamp embedded in QR.

Current client behavior:

- parser accepts Unix timestamp in seconds or milliseconds
- parser normalizes it once to Unix milliseconds inside the app
- all client-side TTL checks use the normalized millisecond value

---

### service

Type:
string

Description:
Display label embedded in QR and validated for size by the client parser.

---

## Optional Display Context

Depending on system design, the QR flow may also be associated with metadata returned by the backend after scan, such as:

- service
- location
- description
- brandColor
- expires

These fields should preferably be resolved server-side after scanning, not embedded directly into QR unless there is a clear reason to expose them.

---

## Current Flow

The implemented mobile flow works as follows:

1. Service generates an authorization request
2. Backend creates request state and nonce
3. QR is rendered with request identifier, nonce, expires and service
4. Mobile app scans the QR
5. App extracts requestId, nonce, expires and service
6. App calls POST /api/auth/qr/scanned
7. Backend returns:
   - challenge
   - service context
   - expiration
8. User reviews the request
9. Device signs challenge
10. App sends POST /api/auth/qr/confirm
11. Backend verifies signature and request state
12. Access is granted or denied

---

## Why QR Must Not Contain Secrets

QR codes are visually exposed and easy to:

- screenshot
- photograph
- forward
- cache
- inspect

If a QR contains reusable credentials, interception immediately becomes a direct compromise.

Toqen avoids this by ensuring that QR contains only request context, while final authorization requires:

- valid device key
- valid challenge signature
- explicit user confirmation
- backend verification

---

## Replay Resistance

QR interception is assumed to be possible.

Replay is mitigated by:

- nonce
- request expiration
- server-side request state
- single-use lifecycle
- challenge signing step

A scanned QR is not sufficient to gain access on its own.

---

## Expiration

QR-based requests must be short-lived.

Recommended TTL:
30 to 60 seconds

Requirements:

- expired requests must be rejected
- backend must enforce expiration
- mobile app normalizes QR expiration to Unix milliseconds and should present expiration state clearly

Even if the QR image remains visible, it must become useless after expiration.

---

## Single-Use Lifecycle

Each QR request must have a lifecycle state on the backend.

Recommended states:

- pending
- scanned
- approved
- denied
- expired
- used

Rules:

- once approved and completed, request becomes used
- once denied, request should not be reusable
- once expired, request must be rejected
- repeated scans of completed requests must fail safely

---

## Backend Resolution Model

The QR itself should remain lightweight.

Preferred model:

- QR carries request_id, nonce, expires and service
- backend resolves all full request details
- backend returns challenge and context after app scan

Benefits:

- smaller QR payload
- less exposed metadata
- easier contract evolution
- better control over request validity

---

## Example Minimal QR Payload

toqenapp://auth?request_id=91f2d&nonce=8fa92c1a&expires=1710000000&service=Example%20Service

---

## Example Expanded Backend Response After Scan

{
"ok": true,
"requestId": "91f2d",
"clientId": "client_123",
"projectId": "project_456",
"challenge": "random_challenge_value",
"service": "Example Service",
"location": "Buenos Aires",
"description": "Login approval request",
"expires": 1710000000
}

This response is resolved server-side and is safer than embedding all context directly inside the QR.

---

## What Must Be Verified By Backend

The backend must verify:

- request_id exists
- nonce matches request state
- request is not expired
- request is not already used
- request belongs to expected flow
- signed challenge matches registered public key
- approval state is valid

The client must never be treated as the final source of truth.

---

## Client Parsing Rules

When the mobile app parses a QR:

- reject unsupported schemes
- reject malformed payloads
- require request identifier
- require nonce
- require expires
- require service
- normalize expires to Unix milliseconds exactly once
- do not trust embedded sensitive data
- pass parsed values to backend for verification

The parser must fail safely on invalid or incomplete QR content.

---

## Anti-Abuse Considerations

Recommended protections:

- rate limiting on scan confirmation endpoints
- request expiration enforcement
- request state machine validation
- duplicate request rejection
- audit logging of request transitions

---

## Future Extensions

The QR format can later be extended with optional fields if needed, for example:

- version
- location hint

Example:

toqenapp://auth?request_id=91f2d&nonce=8fa92c1a&expires=1710000000&service=Example%20Service&v=1

Such extensions should remain non-sensitive and backward-compatible.

---

## Summary

The Toqen QR format is intentionally minimal.

It is designed so that:

- QR codes can be displayed publicly for a short time
- interception alone is insufficient for access
- authorization always requires backend verification
- final approval requires a device-bound cryptographic proof

This keeps QR practical for real-time access while preserving security.
